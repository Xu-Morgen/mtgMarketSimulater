# 项目协作文档索引

> 目的：为多人或多 Agent 协作建立单一事实来源与同步规则，避免相同决策、状态或验收结果在不同文档中重复维护后发生漂移。

## 使用规则

1. 开始实现前，先阅读本索引和与改动范围匹配的权威文档。
2. 在同一变更内更新被影响的权威文档；其他文档只添加链接或简短引用，不复制完整规则。
3. 记录已验证事实时附稳定标识和日期，例如 `I12B`、`I12F`、`AT-03`、`ADR-001`、迁移号、规则版本或任务类型。
4. 尚未验证的内容标记为“待决”或“计划”，不得勾选实施计划或写成已完成。
5. 本索引只维护文档职责与同步机制，不复制各文档的业务内容。

## 代码与文档的同步矩阵

| 发生的变更 | 同步动作 |
| --- | --- |
| 新增/变更 API、DTO、错误码或幂等行为 | 更新 `packages/contracts`、`apps/api/docs/api/`、相关前端 API 调用和测试；若影响需求，更新前/后端需求。 |
| 新增/变更可结算规则 | 更新 `packages/rules` 与规则测试；记录规则版本；同步后端需求、验收定义和受影响迭代 checklist。 |
| 新增表、迁移、事务、不变量或异步任务 | 更新 API 架构、后端需求、运维文档、集成测试和实施计划；重大取舍新增 ADR。 |
| 新增或改变页面、状态归属、交互流程 | 更新 Web 架构、前端需求、主流程验收和端到端测试。 |
| 新增/变更 AI Provider、Schema、预算、降级或权限 | 更新 AI 架构、AI需求、运维文档、测试和 I28AI/I29AI/I32AI checklist。 |
| 变更部署、环境变量、数据卷、备份或监控 | 更新 `.env.example`、README、运维文档、技术边界和 I31B/I32B checklist。 |
| 完成迭代项或验收项 | 先保存测试/演示证据与必要的运维说明，再勾选实施计划中的对应项；同步主流程验收的执行状态；完成整个迭代时同步创建或更新 `progress/<迭代ID>.md`。 |
