# 前端需求

本文件是前端功能、交互、页面状态与人工验收的唯一事实来源；架构选型与完整边界见[技术栈与模块职责边界.md](技术栈与模块职责边界.md)。前端只展示服务器真相和提交用户意图，不结算经济数据。

## 1. 技术与工程要求

- Next.js App Router + React + TypeScript；Tailwind CSS；TanStack Query；Zustand 仅存 UI 临时状态；ECharts 与 Framer Motion。
- 复杂业务表单采用 React Hook Form + Zod 做即时交互校验；Zod 客户端校验只改善体验，Fastify 边界校验和服务端授权仍是最终判断。相关依赖在 I06F 落地时加入，不提前引入第二套 DTO。
- `app/` 使用公开、玩家和管理员路由组组合 `pages/` 页面模块；管理员布局提供独立导航。前端路由保护只用于避免错误展示，`/v1/admin/*` 的服务端角色检查始终是权限权威。
- 所有 API 类型来自共享 `contracts` 包；不得手写与后端漂移的 DTO。所有变更型请求带 idempotency key，成功后以服务端响应更新或失效 TanStack Query 缓存。
- MVP 以桌面端为首要验收环境，并保证登录、开包、NPC/P2P 交易、库存、比赛和管理端关键操作在窄屏下不被阻断；完整移动端适配按产品 I6 在首发后单独验收。
- Vitest/组件测试用于快速反馈，Playwright 覆盖真实浏览器主流程；每个交付页面还必须执行本文件定义的人工验收，不能只以 unit test 通过作为完成依据。

## 3. 通用页面状态与交互门禁

1. 每个数据页面都覆盖首次加载、局部刷新、空数据、失败可重试、无权限和数据过期状态；不能用空白页或永久 spinner 代替明确反馈。
2. 每个 mutation 在提交期间禁用相同动作并显示进度；网络重试复用同一 idempotency key，业务完成或用户重新发起新意图后才生成新键。
3. 金额、费用、保证金、概率、赛果、奖励和下一次可领取时间只显示服务端字段。浏览器可做展示格式化，但不能从其他字段重新推导结算值。
4. 所有价格页面显示数据来源、快照/报价更新时间和过期状态，不使用“Cardmarket 实时价格”表述；无价、零价、映射异常或过期 SKU 显示不可新增交易原因。
5. 表单错误同时支持字段级提示和服务端业务错误；版本冲突、预览过期、余额/库存不足、账户冻结和权限拒绝必须有可恢复操作。
6. 管理端筛选与分页状态写入 URL，刷新和返回后可恢复；发布、停用、冻结、会话撤销及补偿修正必须二次确认，并展示对象、影响摘要和必填原因。
7. 玩家导航不展示管理入口；普通玩家直接输入管理 URL 时显示无权限页，且管理数据请求由服务端返回 403。键盘操作、焦点顺序和表单标签应满足核心流程可用性。

## 4. 管理后台详细需求

### 4.1 登录、首页与权限

- 管理员使用 H1 的账号体系登录，成功后无条件进入 `/admin`；登录 URL 的 `next` 一律忽略，后台布局持续显示当前管理员、环境标识和退出入口。
- 后台首页展示最近失败任务、价格/目录新鲜度、活动状态、待复核异常和最近管理操作摘要；这些均来自只读管理 API。
- 会话过期后保留当前只读筛选条件，登录恢复后重新请求数据；不能缓存管理响应供普通用户查看。

### 4.2 活动发布

- 活动表单至少包含名称、展示文案、活动类型、作用范围、开始/结束 UTC 时间和受服务端规则约束的参数；页面显示本地时区换算，但提交标准化 UTC。
- 发布前必须调用服务端预览，展示匹配范围、参数版本、冲突/上限检查和预计触发任务；预览版本过期时禁止发布并要求重新预览。
- 发布/定时发布、暂停和结束均使用独立幂等 mutation。成功后显示活动版本、操作者、审计记录和 `market.reprice` 任务状态；失败不做乐观伪成功。

### 4.2A 目录同步管理

- `/admin/catalog-sync` 仅管理员可进入；展示 `CatalogSyncStatusDto` 的最近成功版本、当前/最近运行差异、失败摘要与最近 `catalog.sync` 任务。外部下载地址、原始 Provider 响应及内部配置不显示给浏览器。
- 触发同步和重试失败任务均须二次确认，并为每个新意图生成幂等键；任务等待或执行时禁用重复触发，同一意图网络重试复用该键。
- 同步失败时页面明确说明旧目录仍保留，保留任务 ID 以关联服务端审计请求和任务运行历史；轮询只访问本地 `/v1/admin/*` API，普通玩家无导航入口且服务端返回 403。

### 4.3 玩家数据管理

- 支持按用户 ID、用户名、角色、账户状态检索，详情页只读展示账户与会话状态、余额/账本摘要、库存/流水、订单、比赛和最近审计记录。
- 冻结/解冻与会话撤销必须显示影响范围和服务端返回状态。补偿修正必须选择原因分类、填写说明、预览修正后的服务端结果并二次确认。
- 修正余额或库存时只提交补偿命令，不接收“最终余额/最终库存”自由编辑；页面同时展示新生成流水与原始关联记录，确保可追溯。

### 4.4 日志查看

- 审计日志、任务运行日志、异常交易和 Agent 记录采用分页查询；默认最近优先，可从用户、活动、任务或请求详情跳转到关联日志。
- 日志详情显示时间、级别/结果、操作者、动作、实体、请求 ID、幂等键、规则/配置版本及脱敏前后值；不同日志来源应有清晰标签。
- 日志页面只读。复制请求 ID 等排障字段不应复制隐藏敏感信息；任何后端返回的脱敏标记需原样展示。

| --- | --- | --- |

## 6. 自动化与迭代完成条件

- FE-AUTH 至 FE-TOURNAMENTS 的关键成功/失败路径使用 Playwright；金额、权限、幂等和重复点击仍由 API/SQLite 集成测试做权威断言。
- FE-ADMIN-SHELL、FE-ADMIN-EVENTS、FE-ADMIN-USERS、FE-ADMIN-LOGS 覆盖管理员成功路径、普通玩家 403、会话过期、版本冲突、参数越界、重复提交和脱敏显示。
- 一个用户可见迭代只有在页面功能、通用状态、相关 Playwright 和人工验收记录同时完成后才算通过；单元测试或 API 测试通过不能替代浏览器验收。
