# AI 需求

AI 用于增强比赛与 NPC 的真实感，绝不成为经济或比赛结算的权威。完整架构见[技术栈与模块职责边界.md](技术栈与模块职责边界.md)。

## 1. 技术要求

- 使用 OpenAI Responses API 的 Structured Outputs，由 SQLite 持久化任务表和进程内任务循环异步调用。
- 服务器保存 API 密钥与模型配置；浏览器不能调用 OpenAI，也不能获得密钥。
- 每场比赛使用 `tournamentId + narrativeVersion` 去重；设置超时、有限重试、用户日限额和全局预算。

## 2. 输入与输出契约

允许输入：已结算赛果、赛制、卡组特征标签、关键回合标签、NPC 人设、市场氛围与语言偏好。

禁止输入：密码、联系方式、API 密钥、未结算结果、可执行经济操作的指令。

输出必须符合以下 JSON 概念结构：

```json
{
  "headline": "string",
  "summary": "string",
  "highlights": ["string"],
  "npc_quote": "string",
  "tone": "victory | defeat | tense | neutral"
}
```

服务端须校验 JSON Schema、字段长度、语言、敏感内容与赛事 ID；不合格输出不得展示为 AI 战报。

## 3. 权限边界

AI 不拥有函数工具、数据库写入工具、网络搜索工具，不能访问或改变：

- 赛果、随机种子、比赛奖励与成就资格。
- 货币、库存、开包概率、市场价格、订单、保证金。
- 管理后台、用户资料与任何密钥。

AI 返回文本只能在赛果写入后显示，不能作为后端规则输入。

## 4. 降级与验收

- 超时、限流、结构错误、内容过滤或预算耗尽时，保存失败原因并使用模板战报与预置 NPC 台词。
- AI 成功与失败两种路径的赛果、余额、库存、奖励与成就必须完全一致。
- 记录模型、输入摘要哈希、输出、延迟、令牌与成本；仅管理员可查看调试记录。
