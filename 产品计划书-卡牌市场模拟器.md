# 卡牌市场模拟器：产品计划书（MVP 范围已分级）

## 1. 产品概述

### 产品定位

一款可快速部署到服务器的网页游戏。玩家通过初始资金、购买补充包、开包获得卡牌、在模拟市场中买卖卡牌、管理库存，并根据市场价格的上涨与下跌制定经营策略。游戏重点是“收藏乐趣 + 市场交易 + 风险决策”，不涉及现实货币或真实卡牌交易。

### 目标用户

- 喜欢集换式卡牌、开包和收藏的玩家。
- 喜欢价格波动、低买高卖和经营模拟玩法的玩家。
- 希望在小型社群或个人服务器中游玩的用户。

### 核心体验

1. 获得资金并选择购买补充包或直接从市场购卡。
2. 开包获得不同稀有度、属性和价值的卡牌。
3. 根据价格历史与市场信号决定买入、持有或卖出。
4. 在库存、现金流、收藏完成度和风险之间取得平衡。

## 2. 首发版本边界建议

首发版本以单人可玩与 NPC 做市为基础，同时提供受限的玩家双边委托、撮合、挂售确认和模拟履约保证金；不引入现实支付、提现、实体卡牌物流或聊天。这样即使低在线也能完成交易闭环，并将多人交易的反作弊范围控制在订单、资金/库存预占与模拟履约层面。

### E. 市场价格与走势系统

| 编号 | 需求 / 功能 | 说明 | 分类 |
| --- | --- | --- | --- |
| E1 | MTGJSON 卡牌与价格数据导入 | 使用 MTGJSON 的卡牌标识、系列资料及 `AllPricesToday` / `AllPrices` 文件导入可交易卡牌与价格；不直接抓取 Cardmarket 或 TCGplayer 网页。 | 必须 |
| E2 | Cardmarket 欧元参考价 | 以 MTGJSON 提供的 Cardmarket EUR Trend Price 作为欧服市场的外部参考基准；按印刷版本、普通/闪卡等可区分的数据字段映射。缺失价格的卡牌不可交易或采用后台配置的兜底价。 | 必须 |
| E3 | 每日同步与数据快照 | 每日拉取一次 MTGJSON 数据，校验版本/校验和后写入不可变价格快照；失败时继续使用最近一次成功快照，并记录异常。游戏不承诺逐笔实时价格。 | 必须 |
| E4 | 游戏内报价计算 | 当前游戏报价 = 最新外部参考价 × 游戏内市场系数；市场系数由供需、系列周期、事件和流动性共同决定，并保留计算原因。 | 必须 |
| E5 | 上涨与下跌趋势 | 以连续的每日参考价快照形成真实外部涨跌；游戏内系数只做有限扰动，避免价格脱离参考市场或纯随机跳价。 | 必须 |
| E6 | 历史价格曲线 | 分别展示“Cardmarket 参考价”“游戏内成交/报价”两条曲线，支持 7 天、30 天和全部可用历史，并标注数据更新时间。 | 必须 |
| E7 | 基础市场事件 | 系列阶段、赛事结果或运营事件可在配置上限内影响指定卡或系列；事件只改变游戏内市场系数，不覆盖外部参考价快照，并须记录生效区间与原因。 | 必须 |
| E8 | 供需影响价格 | 玩家开出、购买和出售只影响本服务器的游戏内市场系数与流动性，不会回写或影响 Cardmarket 市场。 | 必须 |
| E9 | 系列周期 | 新系列热度、旧系列停印、再版等阶段性机制以有限系数影响游戏内报价；外部价格仍以每日快照为准。 | 必须 |
| E10 | 卡牌关联性 | 同主题、同稀有度或同系列卡牌可联动调整游戏内市场系数，但须设上下限。 | 必须 |
| E11 | 市场指数 | 展示 Cardmarket 参考指数与本服游戏指数，并按系列、稀有度和时间段比较涨跌。 | 必须 |
| E12 | 价格提醒 | 卡牌到达目标游戏内报价、外部参考价涨跌超过阈值或同步失败时，在游戏内提醒。 | 可以 |
| E13 | 行情预览 / 分析提示 | 基于已公开的历史快照和游戏事件给出风险提示，不承诺预测结果。 | 可以 |
| E14 | 市场崩盘与泡沫 | 少数高风险事件可在受配置上限约束下造成游戏内价格大幅变化；外部参考价仍如实保留。 | 可以 |

| 货币 | 仅使用游戏内虚拟货币，不支持充值、提现或现实货币交易。 | 必须 |
| 时间 | 每个现实自然日刷新一次 MTGJSON 外部参考价、发放一次可领取的工作资金并刷新每日比赛；单人模式推进的是游戏内日期和事件，不触发虚假的外部实时价格或重复发钱。 | 必须 |
| 抽卡 | 概率由服务端生成并公开展示；MVP 不启用保底，未来若启用必须先补充版本化规则、状态迁移、事务与重放测试。 | 必须 |
| 市场 | 首发由 NPC 充当做市商；收购价低于卖价，并在玩家市场无订单时提供可买、可卖的基础流动性。 | 必须 |
| 价格 | Cardmarket EUR Trend Price（经 MTGJSON 每日同步）为外部锚点；游戏内报价在锚点上叠加有上限的供需、流动性、系列与事件系数。 | 必须 |
| 计价 | 外部 EUR 参考价先转为整数欧分，再按版本化兑换比例换算为虚拟货币最小单位；兑换比例、舍入方式、最低报价和费用规则均由服务端版本化配置。 | 必须 |
| 交易 | 余额、冻结资金、库存、订单、费用、保证金与相关流水在一次短数据库事务中完成更新。 | 必须 |
| 卡池与图像 | 卡池资料取自 Scryfall Bulk Data，并在本地保存项目所需的卡池和图片缓存；不得逐卡高频请求 Scryfall。外部价格仍取自 MTGJSON 快照。商业公开前须单独完成万智牌名称、卡图与相关商标的使用授权审查。 | 必须 |
| AI 内容 | AI 只生成赛事叙事、NPC 对话和市场氛围文本；所有胜负、奖励、货币、库存、开包与订单均由可审计的服务端规则结算。 | 必须 |

## 5. 建议的数据实体

- 用户：账号、权限、余额、等级、创建时间。
- 卡牌系列：名称、主题、发售状态、市场参数。
- 卡牌：Scryfall ID、系列、稀有度、变体、规则文本、图像链接、MTGJSON UUID、Cardmarket 商品标识、图片与说明。
- 补充包：价格、卡位、概率表、可用时间；MVP 不保存保底状态。
- 玩家库存：用户、卡牌、数量、平均成本、品相（如启用）。
- 外部价格快照：卡牌、数据源（初始为 MTGJSON/Cardmarket）、价格类型、币种、普通/闪卡版本、快照日期、导入版本与校验和。
- 市场报价：卡牌、参考价、游戏内系数、买入价、卖出价、更新时间、流动性、计算原因。
- 价格历史：卡牌、时间点、外部参考价、游戏内报价、变化原因与关联事件。
- 交易流水：用户、类型、数量、价格、手续费、前后余额。
- P2P 委托与成交：方向、用户、卡牌 SKU、原始/剩余数量、限价、状态、买方预占资金、卖方锁定库存/预占保证金、确认时间、成交时间、待履约资金/库存、模拟履约期限、取消原因与保证金扣除记录。
- 开包记录：用户、补充包、消耗、产出卡牌、结果价值。
- 每日工作资金：用户、自然日、可领取金额、领取时间、发放规则版本与幂等键。
- 卡组：用户、赛制、主牌/备牌、锁定库存、合法性校验结果与强度快照。
- 比赛：赛制、报名条件、对手、随机种子、规则版本、结算结果、奖励与 AI 战报状态。
- 成就：用户、成就定义、解锁时间、进度、展示物和奖励流水。
- Agent 生成记录：赛事、输入摘要、模型、结构化输出、失败原因、回退文本、令牌/成本指标与审核状态。
- 市场事件：作用范围、持续时间、影响系数、展示文案。
- 运营活动：类型、标题、展示文案、作用范围、生效/结束 UTC 时间、参数版本、状态、发布人、发布时间和关联重定价任务。
- 管理审计日志：操作人、对象、脱敏前后值、请求 ID、幂等键、时间、来源和结果。

## 6. 页面与导航草案

1. 首页 / 仪表盘：余额、净资产、今日盈亏、市场指数、今日工作资金、今日比赛与待领取奖励。
2. 补充包商店：补充包列表、概率、价格、购买与开包入口。
3. 开包结果：卡牌揭示、重复卡标记与保留、价值统计、再次开包。
4. 市场：搜索、筛选、NPC 买卖报价、玩家买单/卖单、双边订单簿、参考价与游戏内价双曲线、数据更新时间、基础市场事件。
5. 我的库存：持仓、市值、收益率、指定数量/全部持有出售、收藏入口。
6. 卡牌图鉴：系列进度、卡牌详情、未获得卡牌。
7. 卡组与比赛：构筑卡组、报名、对局战报、赛事记录、称号与成就。
8. 成就：目标、进度、展示物与奖励；每日/每周任务留待后续迭代。
9. 经营报表：资金曲线、开包盈亏、交易盈亏、比赛成绩、持仓分布。
10. 管理后台（管理员）：后台首页；内容/概率/市场配置；活动草稿、预览与发布；玩家查询、冻结/解冻、会话撤销及补偿修正；任务与 Agent 记录；可筛选的只读审计/运行日志。

## 7. 验收标准示例

- 玩家能从新存档开始，获得初始资金，购买并开出补充包，且获得的卡牌进入库存。
- 玩家能按当前报价买入和卖出卡牌；余额、库存和交易记录保持一致。
- 卖家挂售前必须确认版本、数量、价格与预计到手金额；成交后取消模拟履约会按规则扣除履约保证金，所有状态和扣款均可追溯，且界面明确不涉及实体卡牌物流。
- 玩家每天只能领取一次工作资金，且可用库存构筑合法卡组参加比赛；比赛成绩、奖励与成就均可审计。
- AI 赛事文本在结构校验通过时显示个性化战报；调用失败时自动显示模板战报，且不改变已由规则引擎结算的胜负与奖励。
- 至少能查看每张卡的 Cardmarket 参考价、游戏内报价与历史走势，并能看出涨跌和数据更新时间。
- 系统每天能成功导入 MTGJSON 价格快照；同步失败不会清空既有价格，并能向管理员记录异常。
- 市场价格会在外部参考价基础上按既定规则更新；事件只能改变游戏内系数，不能篡改外部价格快照。
- 服务可根据部署文档在一台常见 Linux 服务器上启动，重启后数据仍存在。
- 管理员能在不改代码的前提下启停同步卡牌、创建有明确来源的测试/原创例外、配置补充包和基础市场事件。
- 管理员登录后能进入独立后台，发布或暂停一个有生效区间和作用范围的活动，并从只读审计日志追溯预览、发布、重定价和结果；普通玩家不能访问该页面或接口。
- 管理员能检索玩家并查看经济与业务摘要，冻结/解冻账户、撤销会话；余额或库存修正只能走带原因和二次确认的补偿流程，且原始流水保持不变。

## 8. 范围变更规则

- `必须` 项是 MVP 发布门禁；删除、降级或改写时必须同步更新实施计划、主流程验收和相关前/后端需求。
- `应该`、`可以` 项默认不阻塞 MVP；若某个必须项依赖它，应先把依赖范围提升为必须或缩小必须项，禁止只实现无执行能力的配置入口。
- `不需要` 项不得在 MVP 中暗含状态、规则或后台配置；未来启用时按新增需求重新建立规则版本、迁移、契约、测试和验收。
- 范围决策使用稳定需求编号，并在《完整项目迭代实施计划与检查清单》的追踪表中登记主责迭代和证据。
